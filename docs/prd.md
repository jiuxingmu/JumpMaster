# JumpMaster 产品需求文档（PRD）

**文档性质**：从 Demo 到可上架的单一产品事实来源（SSOT）。  
**适用范围**：功能范围、体验标准、阶段目标与上架门禁；技术栈与模块拆分以工程代码与团队规范为准。  
**版本**：1.0（文档）  
**最后更新**：2026-05-14  

---

## 1. 背景与问题

JumpMaster 是一款以**手机摄像头 + 视觉算法**辅助完成**跳绳自动计数**的应用，当前工程已具备多模块雏形（开始 / 历史 / 我的、CameraX、MediaPipe 姿态、Room、域内计数逻辑、音效与播报等），整体仍处于 **Demo 向可发布产品过渡** 的阶段。

**要解决的用户问题**：在家或楼道跳绳时，用户希望**少分心、少操作**，练完能**可信地知道跳了多少次**，并能**简单回看记录**。

**非目标（1.0 明确不做，避免范围蔓延）**

- 不宣称医疗诊断或治疗功效。
- 不默认承诺「竞技级、裁判级」精度；定位为**健身辅助工具**，以可解释、可调节、可纠错为准。
- 不将「传感器装口袋 + 前台服务常驻」等重合规/重电量能力绑死为 1.0 必选项（可作为 1.x 增强单独立项）。

---

## 2. 产品定位

| 维度 | 说明 |
|------|------|
| 核心用户 | 在家或近场所有限空间跳绳、习惯用手机记录训练的普通健身用户。 |
| 核心价值 | **少操作、看得见、能复盘**——开始即练、实时反馈、练完有记录。 |
| 主路径 | **手机 + 摄像头**自动计数；横竖屏与摆放策略需在 1.0 内统一并写清。 |
| 差异化（对内） | 不依赖外接跳绳硬件即可计数；强调**引导 + 可调灵敏度 + 手动纠错**建立信任。 |

---

## 3. 用户与场景

**主要场景**

1. 用户打开 App → 进入「开始」→ 按引导摆放手机 → 开始跳绳 → 实时看到次数 → 结束训练 → 在历史中查看本次与往日记录。
2. 用户从历史页回看最近或某次训练的时长与次数（必要时删除单条记录）；会话文字备注、按周聚合视图等列入 1.x，当前以子 PRD 为准。

**关键约束**

- 光线、距离、全身入镜对视觉计数影响大，产品必须通过**状态提示与文案**管理预期。
- 前后台、锁屏、来电等系统中断需有**明确行为定义**（暂停、停止或提示不支持），避免「以为还在计」的误解。

## 4. 信息架构与功能范围（1.0）

与当前应用结构对齐：**底部三 Tab——开始、历史、我的**。

### 4.1 开始（训练）

| 能力 | 说明 | 阶段 |
|------|------|------|
| 相机预览与权限 | 场景化说明；拒绝权限时的降级与再次引导。 | P0 |
| 人体/画面质量提示 | 入画、距离、光线等可理解状态（文案 + 简单 UI）。 | P0 |
| 开始 / 暂停 / 结束 | 状态清晰；结束页展示次数、时长及一项**质量信号**（如稳定度或有效检测占比，二选一避免过度工程）。 | P0 |
| 计数可信度 | 在域逻辑基础上覆盖典型误检：半身、大幅非跳绳动作、多人同框等（优先提示「单人全身入镜」）。 | P0 |
| 手动纠错 | **结束页** **+1 / -1** 纠偏并写回会话（子 PRD C-05，P0）；**训练中**实时纠偏为增强项，另子 PRD，非当前索引内必达。 | P0 |
| 首次引导 | 首次启动：权限理由 + ≤3 步图文「如何摆放手机」。 | P1 |
| 训练体验增强 | 大屏数字、弱光提示、可选倒计时开始；设置项即时生效或可预期说明。 | P1 |

### 4.2 历史

| 能力 | 说明 | 阶段 |
|------|------|------|
| 列表与详情 | **列表**按时间倒序；**详情**含次数、时长与日期时间。按日/周分组、图表与备注编辑列入 1.x / 另子 PRD（当前 H-01/H-02 为平面列表 + 详情）。 | P1 |
| 删除与清空 | 单条删除、清空全部（二次确认）。 | P1 |
| 空状态 | 引导用户完成第一次训练。 | P1 |

### 4.3 我的（设置与合规入口）

| 能力 | 说明 | 阶段 |
|------|------|------|
| 音效 / 语音播报开关 | 默认合理；开关立即生效或可预期。 | P1 |
| 计数灵敏度 | 档位或滑杆；与引导文案一致。 | P1 |
| 关于 | 应用版本；**用户协议**与**隐私政策**入口（可链至网页，但必须可用、无死链）。 | P1 |
| 反馈 | 邮箱或表单入口（子 PRD P-05）；若后续提供帮助/常见问题页，可与反馈入口交叉引用。 | P2 |

## 5. 分阶段路线图与门禁

采用**门禁式**阶段：每阶段结束须满足本节「门禁」再进入下一阶段。

### P0 — 可信任 Demo

**目标**：种子用户愿意连续使用数日，不因乱跳数、闪退或不知如何摆放而流失。

**门禁**

- 代表机型清单（建议覆盖高中低与多厂商）上，核心训练路径无 P0 级崩溃；相机与推理资源正确释放。
- 室内正常光、单人全身入镜条件下，与人工计数对比的**误差率**经团队实测后写入发布说明，并在 Demo 范围内可接受。
- 用户无需文档即可完成一次「开始 → 练 → 结束 → 看到结果」闭环。

### P1 — 可日常使用 MVP

**目标**：用户会第二次打开「历史」，设置可信。

**门禁**

- 安装到产生第一条历史记录的路径耗时目标：**≤ 2 分钟**（可实验室或远程配置验证）。
- 历史列表滑动流畅；所有可点控件有明确反馈。
- 训练中切后台等行为与文案一致，无「静默错误计数」的歧义。

### P2 — 可上架 Release Candidate

**目标**：满足应用商店政策与基础口碑风险可控。

**门禁**

- 隐私政策完整披露：数据处理范围、是否上传、第三方 SDK 列表；权限最小化与场景说明一致。
- 商店素材与 App 实际 UI 一致：截图、描述、图标、启动体验无测试文案与死链。
- 无障碍：关键控件可读；字体缩放后主数字仍清晰可读。
- Beta 灰度（建议 50–200 人）关闭全部 P0/P1 级问题。

### P3 — 1.0 公开发布

**目标**：下载后能完成首练，首周留存可观测。

**门禁**

- 核心指标（至少）：完成首次训练率、D1 留存、崩溃率/ANR 是否在团队基线内；定义见第 6 节。
- 评分与客诉响应机制就绪（模板化回复与问题归类）。

## 6. 成功指标（1.0 建议）

| 指标 | 定义方向 | 备注 |
|------|-----------|------|
| 首次训练完成率 | 安装后 24h 内完成至少一次有效训练会话的比例 | 需定义「有效会话」最小时长或最少次数 |
| D1 / D7 留存 | 安装后第 1 / 7 日是否打开 App | 可与本地事件或合规分析方案绑定 |
| 计数误差感知 | 客服/反馈中「计数不准」占比 | 配合灵敏度与引导迭代 |
| 稳定性 | 崩溃率、ANR（若上架 Google Play 则对齐 Play vitals） | 与研发基线一致 |

## 7. 风险与对策

| 风险 | 产品对策 |
|------|-----------|
| 光线/距离导致误差 | 引导 + 灵敏度档位 + 手动纠错；常见问题可在商店说明或后续帮助页补充。 |
| 用户不信任黑盒算法 | 质量信号（一项即可）+ 可调参数说明 |
| 耗电与发热 | 训练中省电模式（降帧/降分辨率）作为设置项纳入 1.x 评估 |
| 商店对健康类表述敏感 | 文案限定为健身辅助，避免疗效承诺 |
| 无账号换机丢数据 | 1.0 可接受；导出/云同步列入 1.x 路线图 |

---

## 8. 需求优先级原则

1. **主路径 > 历史 > 设置 > 锦上添花**。
2. **先正确再炫酷**：动效与主题在计数与闭环稳定之后。
3. **每个版本必须有完整用户故事**，禁止合并半个功能。
4. 新增后台常驻、多模态传感器等能力须**单独合规与电量评审**。
