# 子 PRD → Cursor Agent 投喂约定

面向：**按 `README.md` 顺序、单需求单 chat 实现**时的提示词与流程约定。各子 PRD 文末另有**可复制首条消息**（含对应 ID 与文件名）。

---

## 1. 单次对话的输入结构（推荐）

按顺序组装首条用户消息，效果通常最好：

1. **角色与硬约束**（可压缩为一段）  
   - 实现 Android Kotlin / Jetpack Compose 需求。  
   - 严格遵守仓库 `.cursor/rules/jumpmaster-standards.mdc`（圈复杂度、文件长度、MVVM、Room、Compose 等）。

2. **需求正文**（二选一）  
   - 用 Cursor `@` 引用单个子 PRD 文件（推荐）；或  
   - 将子 PRD 全文粘贴进对话。

3. **可选上下文**  
   - `@` 你已知的相关路径（如 `app/...`、`feature/...`、`JumpMasterNavHost.kt`）；若不确定，可写明「请先在仓库内搜索再改」，减少误改。

4. **显式禁止**（与子 PRD 一致即可）  
   - 不做该子 PRD **Out** 表中的任何项；不借机做同文件未列出的「大重构」。

---

## 2. 依赖与顺序

- **硬依赖**以 `README.md` 表格「依赖」列为准（如 H-01 依赖 C-02）。未满足时：要么新开 chat 先完成依赖项，要么在本 chat **仅**做接口占位并在 PR 中写明「依赖未合，本 PR 不合并」。  
- 标 `*` 的项（与 C-01 无硬依赖）可与主流程并行，但**仍建议**按表顺序减少 UI 合并冲突。  
- **主 PRD**（`docs/prd.md`）管范围与门禁；**子 PRD** 管本 chat 验收细则。二者冲突时以**子 PRD In/Out + 验收**为准，并应回头更新主 PRD（见主文档第 9 节）。

---

## 3. Agent 行为期望（减少返工）

- **最小改动**：只改实现当前子 PRD 所需的文件与逻辑。  
- **验收驱动**：实现前先扫一遍文档内所有「验收标准」编号项，改完后逐条对照。  
- **文案与资源**：新增用户可见字符串优先走 `strings.xml`（或项目既有约定）。  
- **Room / IO**：数据库与磁盘访问不阻塞主线程；经 Repository / Dao，与既有架构一致。  
- **说明义务**：若在实现中必须二选一（子 PRD 已写「实现时锁定」的），在 PR 描述用一两句话写清选项。

---

## 4. 本地验证（有则执行）

在仓库根目录：

```bash
./gradlew :app:assembleDebug
```

若子 PRD 涉及数据层或纯逻辑，可补充运行模块内单元测试（若存在）：

```bash
./gradlew testDebugUnitTest
```

具体模块名以工程 `settings.gradle.kts` 为准；Agent 若发现命令失败，应修正命令或说明环境限制，**勿在未跑通的情况下声称完成**。

---

## 5. 合并说明（PR / commit）模板

首行固定引用子 PRD 文件名，便于追溯：

```text
PRD: counter-02-persist-session-on-end.md

- 摘要：一句话说明做了什么。
- 验收：如何对照子 PRD 第 4 节自测（可列步骤）。
```

---

## 6. 何时拆成多个 chat

- 当前子 PRD 验收已全部满足，但你想「顺手」做下一项 → **新开 chat** 并贴下一文件，避免上下文污染与范围说不清。  
- 单 chat 内已大改且子 PRD 未要求 → 应回滚或拆 PR，保持「一文档一 PR」可审性。

---

## 7. 主 PRD 何时需要一起贴

- 讨论**优先级、阶段 P0/P1、是否做 1.x**、商店门禁等 → 附加 `@docs/prd.md` 相关章节即可，**不必**每次全贴主文档。
